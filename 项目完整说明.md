# 企业内部智能助手平台 设计文档

## 1. 项目概述
本项目是一个面向企业员工的智能对话与自动化平台，旨在通过自然语言交互简化企业内部事务处理。员工可通过统一对话接口完成请假申请、会议室预约、个人信息查询等日常操作，提升工作效率。平台集成阿里云百炼大模型（处理复杂语义）及本地Ollama模型（处理高频简单请求），利用Spring AI框架实现多轮对话和Function Calling，通过Spring Security保障数据安全与权限隔离。

## 2. 系统架构
系统采用前后端分离的微服务架构（单体应用亦可），核心模块如下：
- **接入层**：RESTful API网关，处理客户端请求，统一认证与路由。
- **对话引擎**：基于Spring AI，封装大模型调用、多轮会话管理、Function Calling执行。
- **业务服务层**：处理具体业务逻辑（请假、预约、查询等），与数据库交互。
- **数据层**：MySQL持久化业务数据，Redis缓存会话上下文及热点数据。
- **安全层**：Spring Security实现JWT认证与RBAC授权。

交互流程：
1. 用户通过客户端发送自然语言请求至`/api/chat`接口。
2. 对话引擎从Redis加载当前会话上下文，构造消息列表。
3. 根据策略选择模型（阿里百炼或Ollama），调用其API。
4. 若模型返回function call，引擎解析并调用对应业务服务，将结果反馈给模型生成最终回复。
5. 对话上下文更新并持久化至Redis（设置过期时间）。
6. 最终回复返回客户端。

## 3. 数据库设计
### 3.1 实体关系图
- **员工**（employee）: 系统用户，包含基本信息及角色。
- **请假申请**（leave_application）: 员工发起的请假记录。
- **会议室**（meeting_room）: 会议室基本信息。
- **预约记录**（booking_record）: 会议室预约记录。
- **对话历史**（chat_history）: 持久化对话日志（可选，用于分析或恢复）。

### 3.3 索引建议
- employee：`username` 唯一索引
- leave_application：`employee_id`，`status`
- booking_record：`employee_id`，`room_id`，`start_time`
- chat_history：`employee_id` + `session_id` 复合索引

## 4. 接口设计
### 4.1 认证接口
#### 登录
- **URL**: `/api/auth/login`
- **Method**: POST
- **Request Body**:
  ```json
  {
    "username": "string",
    "password": "string"
  }
  ```
- **Response**:
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "token": "jwt_token",
      "user": {
        "id": 1,
        "name": "张三",
        "role": "EMPLOYEE"
      }
    }
  }
  ```

#### 登出
- **URL**: `/api/auth/logout`
- **Method**: POST
- **Headers**: `Authorization: Bearer <token>`
- **Response**: 200 OK

### 4.2 对话接口
#### 发送消息
- **URL**: `/api/chat`
- **Method**: POST
- **Headers**: `Authorization: Bearer <token>`
- **Request Body**:
  ```json
  {
    "message": "我要请明天一天假",
    "sessionId": "uuid"   // 可选，新会话可不传
  }
  ```
- **Response**:
  ```json
  {
    "code": 200,
    "data": {
      "reply": "好的，已为您提交明天（2025-03-16）的请假申请，等待审批。",
      "sessionId": "会话ID"
    }
  }
  ```

### 4.3 业务数据管理接口（管理员专用）
以下接口需角色为ADMIN方可访问。

#### 查询所有请假申请
- **URL**: `/api/admin/leaves`
- **Method**: GET
- **Params**: `page`, `size`, `employeeId`(可选)
- **Response**: 分页数据

#### 审批请假
- **URL**: `/api/admin/leaves/{id}/approve`
- **Method**: PUT
- **Body**: `{"status": 1, "remark": "同意"}`

#### 会议室管理CRUD
- 略，遵循REST风格。

## 5. 关键流程设计
### 5.1 对话处理流程
1. **接收请求**：从请求头解析用户ID，根据sessionId从Redis获取历史上下文（若无则新建）。
2. **构建Prompt**：将当前消息与历史消息合并，并注入系统提示（包括可用的Function定义）。
3. **调用大模型**：根据业务规则选择模型（如简单请求用Ollama，复杂用阿里百炼）。
4. **处理Function Call**：
   - 若模型返回function_call，根据名称路由到对应业务服务（需校验权限）。
   - 执行服务后，将结果作为新消息追加，再次调用模型生成自然语言回复。
   - 循环直至模型返回最终回复（限制最多3次function调用）。
5. **存储上下文**：更新Redis中的会话上下文（设置TTL=30分钟），可选异步写入MySQL历史表。
6. **返回回复**：将最终回复及sessionId返回客户端。

### 5.2 请假申请流程（Function内部实现）
- 解析参数：开始日期、结束日期、原因。
- 校验：开始日期≤结束日期，不能与现有请假重叠（可选）。
- 插入数据库：状态默认为待审批。
- 返回结构化结果：`{"success": true, "leaveId": 123}`，该结果由模型组织成自然语言。

## 6. 安全设计
### 6.1 认证机制
- 使用JWT进行无状态认证，登录后生成token，有效期8小时。
- Spring Security配置JWT过滤器，从Authorization头解析并验证token，构建Authentication对象。

### 6.2 授权控制
- **基于角色**：使用`@PreAuthorize("hasRole('ADMIN')")`等注解限制管理员接口。
- **基于数据归属**：在业务层通过`SecurityContextHolder`获取当前用户ID，查询时自动添加`employee_id = ?`条件，确保用户只能访问自己的数据。
- Function调用时，同样注入用户上下文，在服务实现中校验权限（例如，请假申请自动关联当前用户ID）。

### 6.3 数据隔离
- 数据库设计中外键关联employee_id，查询强制添加该条件（可通过MyBatis拦截器自动附加）。

## 7. 技术实现要点
### 7.1 Spring AI集成
- 引入`spring-ai-ollama-spring-boot-starter`和`spring-ai-alibaba-starter`（假设有阿里百炼对应starter）。
- 配置多个`ChatClient` Bean，分别指向不同模型。
- 使用`ChatMemory`接口实现Redis存储（利用`spring-ai-redis`扩展或自行实现）。
- 注册Function：通过`@Bean`返回`java.util.Function`，Spring AI自动将其包装为可调用工具。

### 7.2 MySQL会话存储
- 将员工与大模型的上下文对话存储在数据库表中，保障会话的连续性

### 7.3 Function Calling实现
- 定义`ApplyLeaveFunction`实现`Function<ApplyLeaveRequest, ApplyLeaveResponse>`。
- 在`@Configuration`类中注册：
  ```java
  @Bean
  @Description("提交请假申请")
  public Function<ApplyLeaveRequest, ApplyLeaveResponse> applyLeave() {
      return new ApplyLeaveFunction();
  }
  ```
- 调用模型时，通过`.withTools(applyLeave, bookMeetingRoom, ...)`传入。

### 7.4 异常处理
- 全局统一异常处理，返回标准格式。
- Function执行中的业务异常（如会议室已满）应返回明确错误信息，由模型重新解释或提示用户。

## 8. 部署与运维
